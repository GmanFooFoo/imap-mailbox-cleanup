# mailbox-cleanup — Design

**Status:** Draft
**Author:** German Rauhut
**Date:** 2026-05-04
**Topic:** Hybrid CLI + Claude Code Skill to triage and clean up an IONOS IMAP mailbox.

---

## 1. Context and Goal

The IONOS mailbox `german@rauhut.com` has accumulated thousands of messages over years: newsletters, GitHub notifications, bounces, large attachments, duplicates, and old archive material. Manual cleanup in any mail client is slow and error-prone.

**Goal:** A repeatable, safe, semi-autonomous cleanup workflow that:

1. **Discovers** what is in the mailbox (categories, top senders, age distribution, attachment sizes)
2. **Acts** on the discoveries iteratively (delete, archive, unsubscribe, dedupe, strip attachments)
3. Is **safe by default** — every destructive operation is dry-run unless explicitly applied
4. Is **auditable** — every applied action is logged so deletions can be reconstructed

**Non-goals:**

- Not a generic mail server admin tool. IONOS IMAP only, single mailbox.
- Not a replacement for a mail client. No reading, no replying, no composing.
- Not multi-user. Single mailbox, single user (German).
- Not a long-running daemon. On-demand operation, manually triggered (later possibly via Marvin cron).

---

## 2. Architecture

```
Claude Code Session
  ↓ /mailbox-cleanup or natural request
Claude Skill (Markdown, Orchestrator)
  ↓ subprocess + JSON
CLI: mailbox-cleanup <subcommand> [--apply | --json]
  ↓ imap-tools
IONOS IMAP (german@rauhut.com)
```

**Separation of concerns:**

- **CLI** is dumb and atomic: one subcommand executes one operation. No state between calls. No interactive prompts. Output is either human-readable text or `--json`.
- **Skill** is the brain: runs `scan`, parses JSON, presents to user as Markdown, asks which operations to run, sequences subsequent calls, always shows dry-run preview before `--apply`.

**Why hybrid (not just CLI, not just Skill):**

- Pure CLI would force the user to remember subcommand combinations and read JSON
- Pure Skill (no CLI) would force IMAP logic into Markdown / Claude tool calls — fragile, slow, hard to test
- Hybrid: CLI is testable and reusable (Marvin can call it later); Skill provides the conversational UX

## 4. Credentials

**Storage:** macOS Keychain via the Python `keyring` package. Two entries:

- Service: `mailbox-cleanup`, account: `german@rauhut.com`, password: IONOS password
- Service: `mailbox-cleanup`, account: `imap-server`, password: `imap.ionos.de`

**Setup:** `mailbox-cleanup auth set` runs an interactive prompt and writes both entries.

**Why Keychain (not .env, not 1Password CLI):**

- macOS-native, encrypted at rest, no plaintext file
- No additional dependency on 1Password CLI binary
- `keyring` is a single Python package, no subprocess overhead

**Connection params:** hardcoded for IONOS (`imap.ionos.de`, port 993, SSL). Not generalized — single-mailbox tool.

## 5. Safety

**Dry-run by default.** Every destructive subcommand (`delete`, `move`, `archive`, `unsubscribe`, `dedupe`, `attachments --strip`, `bounces`) does **not** modify the mailbox unless `--apply` is passed. Default output without `--apply`: a list of UIDs and a summary "would affect N messages."

**Audit log.** Every `--apply` call appends to `~/.mailbox-cleanup/audit.log` with:

- ISO timestamp
- Subcommand and arguments
- Folder
- List of affected UIDs
- Result (success / partial / failure with error)

Format: one JSON object per line (jsonl). Allows post-hoc reconstruction via `grep` / `jq`.

**No undo.** IMAP `EXPUNGE` is irreversible. Skill always shows preview + asks for confirmation before `--apply`. The audit log is the only safety net post-deletion.

**Soft-delete preference.** `delete` defaults to **moving to Trash folder** (`Papierkorb` on IONOS — German default), not `EXPUNGE`. A separate `purge-trash` command (future, not v1) can hard-delete from Trash. This means recent mistakes are recoverable from the IONOS web client for the IONOS-defined Trash retention period.

## 6. Subcommands

| # | Subcommand | Purpose | Required args | Dry-run by default |
|---|------------|---------|---------------|---------------------|
| 1 | `auth set` | Interactive: write IONOS credentials to Keychain | — | n/a |
| 2 | `auth test` | Connect, list folders, disconnect | — | n/a |
| 3 | `scan` | Discovery — classify INBOX, return report | `--folder=INBOX` (default) | n/a (read-only) |
| 4 | `senders` | List top-N senders with counts | `--top=50`, `--folder=INBOX` | n/a (read-only) |
| 5 | `delete` | Move messages matching filter to Trash | one of `--sender=` / `--subject-contains=` / `--older-than=` | yes |
| 6 | `move` | Move messages matching filter to target folder | `--from-filter=...`, `--to=Folder` | yes |
| 7 | `archive` | Bulk: messages older than N → `Archive/YYYY` | `--older-than=12m` | yes |
| 8 | `unsubscribe` | Parse `List-Unsubscribe` header for sender, execute (HTTPS POST or `mailto:`) | `--sender=` | yes |
| 9 | `dedupe` | Find Message-ID duplicates, keep oldest | `--folder=` | yes |
| 10 | `attachments` | Find / strip large attachments | `--size-gt=10mb`, `--older-than=6m` | yes |
| 11 | `bounces` | Find bounce / auto-reply messages | `--folder=INBOX` | yes |

**Common flags:** `--json` (structured output), `--apply` (execute, default off), `--folder=` (target IMAP folder), `--limit=N` (cap operation size).

**Filter combinability:** `delete --sender=linkedin.com --older-than=3m --apply` works as expected (AND across filters).

**Time syntax for `--older-than`:** `Nd` / `Nw` / `Nm` / `Ny` (days/weeks/months/years). Simple and unambiguous.

**`unsubscribe` semantics:** parses RFC 2369 `List-Unsubscribe` header. Two forms supported:

- `<https://...>` — HTTPS POST per RFC 8058 if `List-Unsubscribe-Post: List-Unsubscribe=One-Click` is present, otherwise HTTPS GET
- `<mailto:...>` — sends an unsubscribe email via SMTP (uses the same IONOS account; SMTP credentials reuse the IMAP password)

If both are present, HTTPS is preferred. `unsubscribe --apply` performs the action **and** moves the matching messages to Trash. Without `--apply`, prints the URL/mailto that would be hit.

## 7. Discovery Report Schema

`scan --json` produces a single JSON object. This is the contract between CLI and Skill.

```json
{
  "schema_version": 1,
  "scanned_at": "2026-05-04T14:23:00Z",
  "folder": "INBOX",
  "total_messages": 14823,
  "size_total_mb": 4187,
  "categories": {
    "newsletters": {
      "count": 3421,
      "top_senders": [
        {"sender": "newsletter@linkedin.com", "count": 892, "has_unsubscribe": true},
        {"sender": "noreply@github.com", "count": 1247, "has_unsubscribe": true}
      ]
    },
    "automated_notifications": {
      "count": 2104,
      "top_senders": [
        {"sender": "notifications@github.com", "count": 1247}
      ]
    },
    "bounces_and_autoreplies": {
      "count": 87,
      "samples": [
        {"uid": 12421, "from": "MAILER-DAEMON", "subject": "Undelivered Mail Returned"}
      ]
    },
    "large_attachments": {
      "count": 142,
      "size_mb": 2890,
      "top_offenders": [
        {"uid": 12421, "subject": "Folien Q3", "size_mb": 47, "from": "kollege@firma.de"}
      ]
    },
    "duplicates": {
      "count": 38,
      "groups": [
        {"message_id": "<abc@x>", "uids": [101, 102, 103]}
      ]
    },
    "old_messages": {
      "older_than_12m": 8421,
      "older_than_24m": 5102,
      "older_than_60m": 1247
    },
    "by_year": {
      "2018": 421,
      "2019": 892,
      "2020": 1421,
      "2021": 1892,
      "2022": 2104,
      "2023": 2421,
      "2024": 2892,
      "2025": 2421,
      "2026": 359
    }
  },
  "recommendations": [
    "1247 GitHub-Notifications older than 6 months → 'delete --sender=noreply@github.com --older-than=6m'",
    "892 LinkedIn messages with Unsubscribe link → 'unsubscribe --sender=newsletter@linkedin.com'",
    "142 attachments over 10 MB take 2.9 GB → 'attachments --size-gt=10mb --older-than=6m'"
  ]
}
```

**Schema versioning:** the `schema_version` field at the top is checked by the Skill. If the version the Skill expects doesn't match what the CLI emits, the Skill refuses to interpret the report and asks the user to update one side.

**Size calculation:** `size_total_mb` and per-message size come from IMAP `FETCH RFC822.SIZE` (server-reported byte count, includes headers + body + attachments). Cheap and reliable — no body download needed for size.

### Classification Rules

| Category | Rule |
|----------|------|
| **newsletter** | `List-Unsubscribe` header present **OR** sender local-part matches `newsletter`, `noreply`, `no-reply`, `news`, `marketing` |
| **automated** | sender local-part matches `notifications`, `bot`, `service`, `alerts`, `system`, `daemon`, `automation` |
| **bounce** | sender is `MAILER-DAEMON` or `postmaster` **OR** subject starts with `Undelivered`, `Returned`, `Mail Delivery`, `Delivery Status Notification`, `Auto-Reply`, `Out of Office`, `Abwesenheits` |
| **duplicate** | identical `Message-ID` header (true dupe) — strict v1; fuzzy matching (From + Subject + Date hash) deferred to v2 |
| **large_attachment** | message size > 10 MB |

A message can be in multiple categories (e.g. a newsletter that's also old and has a large attachment). `scan` reports counts per category independently; it does not assign a single label.

## 8. Skill Flow

The Skill is the conversational orchestrator. Pseudocode for the main loop:

```
1. User invokes /mailbox-cleanup
2. Skill: subprocess `mailbox-cleanup auth test --json`
   - on failure: tell user to run `mailbox-cleanup auth set` in terminal, stop
3. Skill: subprocess `mailbox-cleanup scan --json`
   - parse JSON, validate schema_version
4. Skill renders Markdown summary:
   "Mailbox: 14.823 messages, 4.2 GB
    Top categories:
      1. Newsletters: 3.421 (LinkedIn 892, GitHub 1.247, ...)
      2. Old messages: 8.421 older than 12 months
      3. Large attachments: 142 messages, 2.9 GB
      ...
    Recommendations:
      [1] Delete 1.247 old GitHub notifications
      [2] Unsubscribe from LinkedIn (892 messages)
      [3] Strip 142 large attachments older than 6 months"
5. Skill: "Welche Kategorie / Empfehlung willst du angehen?"
6. User picks (e.g. "1")
7. Skill: subprocess `mailbox-cleanup delete --sender=noreply@github.com --older-than=6m --json`
   (no --apply → dry-run)
8. Skill renders preview: "would move 1.247 messages to Trash, here are 5 examples: ..."
9. Skill: "Apply?"
10. User: "ja"
11. Skill: subprocess `... --apply --json`
12. Skill renders result: "moved 1.247 messages, audit log entry written"
13. Goto 5 (next category)
```

**Skill responsibilities:**

- Schema-version check on every CLI call
- Always show dry-run preview before any `--apply`
- Render JSON to readable Markdown tables
- Translate user intent ("lösch alles von LinkedIn") into specific subcommand calls
- Handle exit codes: 0 = OK, 2 = connection, 3 = auth, 4 = bad args, 5 = partial failure

## 9. Error Handling

### CLI side

| Failure | Exit code | Behavior |
|---------|-----------|----------|
| Connection (timeout, DNS, TLS) | 2 | Single retry with backoff (2s, 4s); JSON error if still failing |
| Auth (invalid creds, no Keychain entry) | 3 | Clear stderr message: "Run `mailbox-cleanup auth set`" |
| Bad arguments | 4 | argparse / click default; usage hint |
| IMAP operation failure on individual UID | — | Log, continue; final summary lists per-UID failures |
| Rate-limit / server-side throttle | — | Backoff (5s, 15s, 60s), then exit 5 with partial-success report |

### Skill side

- On exit ≠ 0: parse stderr or `--json` error block, present human message, do **not** continue silently
- On `--apply` partial failure: show audit log path, "X of Y succeeded, Z failed — see log"
- On schema mismatch: refuse to interpret report; tell user "CLI version newer/older than Skill expects, please update one"

## 10. Testing

### CLI

- **Unit tests** for `classify.py` with `.eml` fixtures (newsletter, bounce, automated, plain) — pytest
- **Integration tests** against a local IMAP server in Docker (`greenmail` or `dovecot`) seeded with fixture mails. Covers connection, scan, delete, move, dedupe end-to-end.
- **Smoke test** against IONOS production: only `auth test` and `scan` (read-only). Manual, not in CI.
- CI: GitHub Actions, runs unit + integration on push.

### Skill

- Manually verified in a Claude Code session — Skills are not auto-testable today.
- The Skill includes a `verify` checklist in its Markdown: "after install, run `mailbox-cleanup auth test`, then `mailbox-cleanup scan --json | jq .total_messages` should return a number."

## 11. Out of Scope (v1)

Captured here so they don't leak into v1 implementation:

- Multi-account support (config file, `--account=` flag)
- Provider abstraction (Gmail, Office365)
- Fuzzy duplicate detection (hash over From+Subject+Date)
- Auto-categorization beyond rule-based (no ML / LLM classification in CLI — Skill can do that on top)
- Marvin cron integration (separate later epic)
- Web UI / TUI (`textual` etc.)
- Inbox-Zero interactive triage subcommand. Note: the user's original brainstorming request included "Inbox Zero Triage" — that is realized by the Skill's discovery → preview → apply loop in §8, **not** by a dedicated CLI subcommand. The CLI stays atomic.
- `purge-trash` (hard delete from Trash) — only soft-delete in v1
- Moving the spec into the project's own README; it stays in `docs/` for history

## 12. Open Questions for Implementation Phase

These are deferred to writing-plans:

1. **CLI framework:** `click` or `argparse`? (lean toward `click` for nested subcommands, but `argparse` is stdlib)
2. **Python version:** 3.11 minimum? 3.12?
3. **Packaging:** `pipx install` vs plain `pip install -e .` in a venv vs `uv tool install`?
4. **Folder name for Trash:** IONOS uses German UI by default. Detect dynamically via `LIST` and look for `\Trash` SPECIAL-USE flag (RFC 6154) — fall back to literal `Papierkorb` / `Trash`.
5. **Folder name for Archive:** same approach — try `\Archive` SPECIAL-USE, fall back to literal `Archive`.
